## 📐 版本號規則（內部）

格式：**`v 大.中.小`**（例：`v1.2.3`）

| 位數 | 何時 +1 | 進位後 | 範例 |
|------|---------|--------|------|
| **大版本** | 開服、引擎/架構級重做、經濟體系整體重置 | 中、小歸 0 | v1 → v2 |
| **中版本** | 新增一個**完整系統或大功能** | 小歸 0 | 轉蛋系統、購物商城整修、完美回溯系統 |
| **小版本** | 數值調整、單一功能、bug 修復、體驗優化 | — | 掉率/機率、藍字移除、單一指令 |

**慣例**：以天為單位，每天一個版本條目；當天最大變更決定進位位數；同日多項併入同條目；新版本加最上方；測試中功能加 `-beta`。

**分類標籤**：🆕 新增　⚖️ 調整　🐛 修復　🧹 優化

### ⚖️ 調整（背包全滿 + 堆疊上限 9999）— 需重編譯+重啟

### 🐛 修復（GM 帳號可以交易了）
- 症狀：GM 帳號點交易跳「管理帳號無法進行」（client 端依登入 Admin byte 自己擋）。

### 🐛 修復（好友/組隊上線狀態沒及時更新）
- 根因：`ServerInitializer.java` 的 Netty pipeline **缺 `IdleStateHandler`** → `MapleServerHandler.userEventTriggered`（送 ping、沒回 pong 就踢）從不觸發 → 玩家閃退/斷網(沒正常登出)時 server 不主動偵測，要等 TCP timeout(數分鐘)才斷線，期間好友/組隊一直顯示在線。
- 修法：pipeline 加 `IdleStateHandler(15,0,0)`（15 秒無封包→送 ping；既有邏輯 30 秒無 pong→斷線）。死連線約 **45 秒內**被偵測並通知好友/組隊。正常登出本來就即時，這修的是不正常斷線。

## v1.2.1 — 2026-05-26（可觀測性 Profile A：伺服器端遙測檢查點）

> 排查 error code 5 (Access is denied) 暴露遙測盲區：崩潰在 client 側、伺服器零紀錄、斷線無法分辨原因、缺崩潰前封包。本版補上**伺服器端**檢查點（Profile A）。完整三層設計（A伺服器/B玩家client/C管理者debug）見計畫文件。

### 🐛 修復（error code 5 真正 root cause — 多段技能 splash 封包越界）
- **症狀**：觀察者(其他玩家)在 **聖魂騎士(KoC, job1111) 施放技能時**崩 error code 5 並斷線；施法者本人不崩；冒險者(盜賊)同樣狂打不崩。
- **root cause**：先前加的 `DamageParse.applyAoeSplash`(群攻補傷/補打擊感) 在群攻技能(mobCount>1)時呼叫 `broadcastSplashVisual`，**廣播假的 `closeRangeAttack` 給全地圖(含觀察者)**，其 `tbyte = (怪數<<4) | 1` ← **硬寫死「每隻怪1下」**。但聖魂騎士 **靈魂突刺 11111006 是 attackCount=4(打4下) + mobCount=3(群攻)** → splash 封包宣告1下、技能實際4下 → **觀察者 client 依技能段數讀傷害陣列時讀超過封包→越界記憶體存取→崩**(error5；ProcMon 無檔案被拒因為是記憶體層級)。
- **排查歷程(三層觀測 + A/B)**：斷線.txt 抓到崩潰前封包、地圖負載抓到物件量、ProcMon 證實 client 0 筆 ACCESS DENIED(非檔案)；A/B 測試(char6冒險者同樣洪水卻不崩)排除「物件洪水」；逐一排除膚色/超等技能(10001000全角色共有)/wz檔(全一致)/KoC職業本身。最終定位到 applyAoeSplash 對多段技能 mishandle(服主假說正確)。
- **修法演進 → 最終(依服主指示)：整個 splash 補傷害機制移除**：
  - 初版「多段技能跳過 splash」(safe)→ 不崩但多段無 splash。
  - 試「完整版」(tbyte 段數＋每怪 damage 筆數都用 `getAttackCount()` 拆段)→ **實測仍崩**(多段 splash 廣播除段數外還有別的位元細節對不上)→ 回退。
  - **最終**：服主決定**整個補傷害機制全部拿掉** → 註解掉 `DamageParse` 兩個 `applyAoeSplash(...)` 呼叫點(近距 line534、魔法 line772)。所有技能都不再有 splash 補打，**徹底不崩**。原生 AOE(技能本身 mobCount)不受影響。要恢復取消那兩行註解即可。
  - ⏳ 若以後要重做多段 splash：需逐位元比對正常多段攻擊封包 vs splash 封包結構。
- **意義**：聖魂騎士(KoC)本身沒問題、修完正常可玩，**不需轉職、不需禁 KoC**。編譯 BUILD_EXIT=0、已安全重啟。

### 🆕 新增（可觀測性，集中於 `MapleServerHandler.java` + 新檔 `ConnDiag.java` + `MaplePacketEncoder.java` + `server/Start.java`）
- **`handling/ConnDiag.java`（新檔）**：每條連線的診斷資料，存在 Netty channel 屬性 `TWMS_CONN_DIAG`。欄位：connectTime、lastRecv(最後收到的封包名)、lastException(連線例外)、sentOpcodes(最後30個送出封包 opcode 環狀緩衝，synchronized)。同一連線的 encode/channelRead/channelInactive 皆在同一 EventLoop 執行緒。
- **斷線原因日誌 `logs/Data/斷線.txt`**：`MapleServerHandler.channelInactive` 在玩家斷線時記 `[完整時間] 帳號 角色(charid) 地圖 原因 在線秒數 + 最後收 + 最後送(舊→新30個)`。原因判定：`exceptionCaught`(原為空)現會把例外存入 ConnDiag→有例外=「異常(類別:訊息)」、loginState 為換頻/商城轉換=「正常」、否則=「登出或未知」。**補上原本完全缺的「崩潰前送了什麼封包 / 為何斷線」節點**。

    UPDATE drop_data d
    JOIN ( SELECT dropperid, COUNT(*) AS n FROM drop_data
           WHERE itemid BETWEEN 1000000 AND 1999999 AND questid=0
             AND dropperid NOT IN (<boss清單>)
           GROUP BY dropperid ) t ON d.dropperid = t.dropperid

### 🚢 全航線「候船室常開」（方案 C）

**需求**：原本各航線在「船/車航行途中」會把玩家擋在候船室門外（顯示「航行中、約 X 分鐘後回站」）。服主選方案 C — **候船室常開**：玩家隨時都能進候船室等下一班，下一班發車時自動載走，檢票員不再擋人。

**原理**：每個 event 腳本用 `em.setProperty("entry", ...)` 控制能否進候船室。週期 = `scheduleNew()` 設 `entry=true`（開放） → `stopEntry()` 設 `entry=false`（發車前關門） → `takeoff()`（把候船室玩家 changeMap 上船、`docked=false`）→ `arrived()`（送下船）→ `scheduleNew()` 重新靠岸循環。6 個檢票員 NPC 的判斷邏輯**完全一致**：`entry=="true"`→放行(sendYesNo)；`entry=="false" && docked=="true"`→「準備開走」擋人；否則(航行中)→「航行中約 X 分鐘後回站」擋人。

**改法**：**只把每個 event 腳本 `stopEntry()`/`stopentry()` 裡的 `em.setProperty("entry","false")` 移除**，使 `entry` 在 `scheduleNew()` 設為 `true` 後**永不關閉**。檢票員因此永遠走第 1 分支放行，後兩個擋人分支變成死碼（不再觸發），無需改動任何 NPC 檔。**船的載客狀態機(scheduleNew/takeoff/arrived 的 changeMap、docked、em.schedule 循環)完全未動**。

- `scripts/event/Boats.js`（魔法森林↔天空之城）：`stopentry()` 移除 `entry="false"`，**保留 `Orbis_Boat_Cabin.resetReactors()`/`Ellinia_Boat_Cabin.resetReactors()`**（船艙反應爐重置屬載客邏輯，不可省）。docked/takeoff/arrived/invasion(巴洛古召喚) 全未動。
- `scripts/event/AirPlane.js`（KC↔CBD 飛機）：`stopEntry()` 清空（原本只設 entry=false）。先前修過的狀態機（scheduleNew 設 entry/docked=true、takeoff 設 docked=false）維持不動。
- `scripts/event/Trains.js`（天空之城↔露比）：`stopEntry()` 清空。docked/takeoff/arrived 未動。
- `scripts/event/Flight.js`（天空之城↔神木村）：`stopEntry()` 清空。docked/takeoff/arrived 未動。
- `scripts/event/Subway.js`（KC↔NLC 地鐵）：`stopEntry()` 清空。docked/takeoff/arrived 與到站/離站廣播未動。

**檢票員（6 個 NPC）**：1032008(Boats)、2012013(Trains)、2082001(Flight)、9201068(Subway)、9270038/9270041(AirPlane) — 皆判斷 `entry`，entry 常開後**自動全部放行**，「準備開走」「航行中約 X 分鐘後回站」兩段文字成為死碼永不顯示，依鐵則「只動擋不擋人」不更動 NPC 檔（留死碼無害）。**無任何 NPC 是只判斷 docked 而擋人**。
